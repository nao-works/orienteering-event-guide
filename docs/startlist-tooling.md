# スタートリスト生成ツールの役割

この repo には Oガイドと Oスタの公開ブラウザ版を置く。

スタートリスト生成には、ブラウザで即時に使える便利さと、大会運営の正本を作るための再現性の両方が必要になる。この二つは同じものではない。

## Oスタ

Oスタは、JOY / Japan-O-entrY 形式の CSV をブラウザで読み込み、その場でスタートリストを作るための簡易ツール。

向いていること:

- 入力 CSV のざっくり確認
- 小規模イベントの下書き
- クラス順やレーンの試行
- WRE / ランキング順スタートの試行
- Mulka2 CSV 形式のたたき台作成

現時点で弱いこと:

- イベント固有ルールの明示的な設定管理
- 入力ファイル、seed、生成条件、警告、出力件数をまとめた manifest
- 重複カード番号、空カード、固定ゼッケン、フリースタート、複数日大会の体系的な検証
- 補正履歴の追跡

したがって、Oスタ単体の出力を大会運営の最終正本として扱わない。

## 運営向け正本生成フロー

大会運営の正本生成は、入力検査、イベント設定、補正履歴、生成記録を残せる運営向けフローを基準にする。

必要な性質:

- JOY 入力の正規化
- YAML などによるイベント固有ルール管理
- corrections / start preferences の記録
- 複数日、複数レーン、フリースタート、固定ゼッケンへの対応
- validate による事前検査
- Mulka CSV、公開 CSV / Markdown の生成
- 生成記録による再現性の確保

現在の内部実装では、このフローを CLI / core workflow として管理している。ただし、オリエンテーリング大会の主催者や運営者は非エンジニアであることが多いため、公開利用者に CLI の理解を求める形にはしない。

この workflow は、参加者データや大会固有の補正を扱うため、公開する場合はサンプルデータと生成済み出力を匿名化してから切り出す。

## 方針

1. Oスタはブラウザ版の簡易・下書きツールとして維持する。
2. 正本生成の仕様は運営向け正本生成フローを基準にする。
3. Oスタを強化する場合は、内部 core と同じ検証観点や fixture を共有する。
4. 内部 CLI を公開する場合は、実名入りサンプルや生成済み出力を含めない。
